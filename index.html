Unit Test


public class BandTest {
 @Test
 public void testBandPlaySomething() throws Exception {
  Band band = new Band(guitarist, singer, drummer);

  RockerGuitarist guitarist = Mockito.mock(RockerGuitarist.class);
  RockerDrummer drummer = Mockito.mock(RockerDrummer.class);
  Singer singer = Mockito.mock(Singer.class);

  when(guitarist.playGuitar()).thenReturn("and I'm Slash, ");
  when(singer.singingSomething(true)).thenReturn("I'm Axl Rose ");
  when(drummer.letsGetBeat()).thenReturn("now, Let's Rock!");

  assertEquals("I'm Axl Rose and I'm Slash, now, Let's Rock!",
						band.playSomething(true));
 }
}

Dagger 2

  • è un framework basato sulla dependency injection
  • facilita lo sviluppo in caso di Unit Test
  • disaccoppia le classi unite tra loro da una relazione
  • aumenta la riutilizzabilità del codice

Dependency Injection

una classe lega la propria relazione con un'altra classe dall'esterno: nessun oggetto istanzia un altro oggetto, questo compito è delegato ad una terza classe di configurazione che fornisce le istanze stesse

Dagger fornisce delle classi "Helper" che espongono un builder statico, il quale si occupa della costruzione delle dipendenze a CompileTime (!), legando i @Module tra loro, invocandone i metodi di providing,(vd. Factory) allo scopo di passare i parametri di cui il @Component ha bisogno per l'@Inject degli oggetti! BOOOOMMM!

Dagger ha la caratteristica, che piace a noi sviluppatori, di sfruttare l'Annotation processing di Java

e non gli xml come altre librerie di dependency injection

- Reflection + Annotation

  • @Module
  • @Component
  • @Inject

@Module

crea le dipendenze attraverso metodi pubblici che istanziano (@Provides) gli oggetti
